# Feature Specification: QR Code Microservice

**Feature Branch**: `018-qr-code-microservice`

**Created**: 2026-07-09

**Status**: Draft

**Input**: User description: "create a specification for extracting the qr code generation out into it's own microservice. Examine the existing app to identify the existing functionality related to qr codes, see how it is used etc. Where possible just extract the code"

## Background

The Go app currently contains `app/internal/qrcode/qrcode.go` with two functions:

- `BuildPlayURL(publicHost, windowID string) string` — constructs the `/play?w=<windowID>` URL a QR code encodes
- `RenderPNG(content string, size int) ([]byte, error)` — encodes a URL as a PNG using `github.com/skip2/go-qrcode`

These are called from two routes in `app/main.go`:

- `GET /qr.png` (`handleQRPNG`) — looks up the current window ID from Redis, discovers the public ngrok host, calls `BuildPlayURL` + `RenderPNG`, and returns the PNG to the browser
- `GET /repo-qr.png` (`handleRepoQRPNG`) — renders a static QR code for the GitHub repo URL with no Redis or ngrok dependency

The extraction goal is to move the QR rendering capability into a dedicated microservice following the same pattern as `commits-service` and `scores-service`. The app service will call the new service over HTTP instead of generating QR PNGs in-process.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - QR Code Rendered by Dedicated Service (Priority: P1)

The presenter opens `/host` and the QR code displayed to attendees is generated by the new
qr-service rather than by the app process. Attendees scan the code and reach the game exactly as
before. The behaviour at the browser is identical; only the internal rendering path changes.

**Why this priority**: This is the core extraction. Everything else is cleanup or operational
concerns. If this works, the feature delivers its value.

**Independent Test**: Start the compose stack with the new qr-service running. Open `/host`,
confirm a QR code image loads. Scan it with a phone and confirm the `/play?w=<id>` URL is reached.

**Acceptance Scenarios**:

1. **Given** the compose stack is running and a window is active, **When** the browser loads `/qr.png`, **Then** a valid PNG image is returned and decodes to the expected `https://<ngrok-host>/play?w=<windowID>` URL
2. **Given** the qr-service container is not running, **When** a request is made to `/qr.png`, **Then** the app returns HTTP 503 (the same error it returns today when Redis or ngrok is unavailable)
3. **Given** no active window exists, **When** the browser requests `/qr.png`, **Then** the app returns HTTP 503 (unchanged from current behaviour — this error originates from Redis, not from qr-service)

---

### User Story 2 - Repo QR Code Migrated (Priority: P2)

The static `/repo-qr.png` route also delegates to the qr-service. The GitHub repo URL is encoded
into a QR code by the service on each request.

**Why this priority**: The app's `handleRepoQRPNG` function contains a direct call to the same
qrcode package. Migrating it completes the extraction and leaves no residual QR rendering code in
the app.

**Independent Test**: Request `GET /repo-qr.png`. Confirm the PNG decodes to the project's GitHub URL.

**Acceptance Scenarios**:

1. **Given** the stack is running, **When** `/repo-qr.png` is requested, **Then** a PNG that decodes to the GitHub repo URL is returned with `Cache-Control: public, max-age=86400`
2. **Given** the qr-service is unreachable, **When** `/repo-qr.png` is requested, **Then** the app returns HTTP 503

---

### User Story 3 - qr-service Works as a Standalone Service (Priority: P3)

A developer can call the qr-service directly (on its own published port) to generate a QR code
PNG for any URL, without going through the app service.

**Why this priority**: Follows the pattern set by `commits-service` (port 8082) and `scores-service` (port 8083). Useful for debugging and local tooling without needing the full stack.

**Independent Test**: `curl -o /tmp/test.png "http://localhost:8084/qr.png?content=https://example.com&size=256"` and confirm a valid PNG is produced.

**Acceptance Scenarios**:

1. **Given** qr-service is running, **When** `GET /qr.png?content=<url>&size=<pixels>` is called directly, **Then** a valid PNG encoding the requested URL at roughly the requested size is returned
2. **Given** the `content` parameter is missing, **When** a request is made, **Then** the service returns HTTP 400
3. **Given** a non-positive or unreasonably large `size` is given, **When** a request is made, **Then** the service applies a sensible default or clamped maximum and returns a valid PNG

---

### Edge Cases

- What happens when the qr-service is temporarily unavailable mid-request to `/qr.png`? The app must not panic; it must surface HTTP 503.
- What happens if `content` URL is very long (e.g., >200 chars)? QR error-correction level Medium can encode long strings but may produce a dense code. The service generates whatever the library produces; this is an acceptable trade-off for the extraction scope.
- What if the requested `size` is 0 or negative? The service defaults to 320 (matching current behaviour) or rejects with 400.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: A new `qr-service` MUST expose `GET /qr.png?content=<url>&size=<pixels>` and return a PNG image encoding `content` at approximately `size×size` pixels
- **FR-002**: If `content` is absent or empty, the service MUST return HTTP 400
- **FR-003**: If `size` is absent, zero, or negative, the service MUST use a default of 320 pixels (matching current app behaviour)
- **FR-004**: The `app` service's `handleQRPNG` handler MUST call `qr-service` to render the PNG instead of calling the in-process qrcode package; the handler remains responsible for looking up the current window ID and discovering the public ngrok host
- **FR-005**: The `app` service's `handleRepoQRPNG` handler MUST call `qr-service` to render the static repo QR PNG instead of calling the in-process qrcode package
- **FR-006**: `app/internal/qrcode/qrcode.go` MUST be removed (or reduced to the URL-building helper only) once the rendering path is delegated to qr-service
- **FR-007**: `qr-service` MUST be defined as a service in `docker-compose.yml` following the same pattern as `commits-service` and `scores-service`
- **FR-008**: `qr-service` MUST be added to nginx's upstream routing so requests from the browser for `/repo-qr.png` could in future be served directly; for now proxying happens app-side
- **FR-009**: `qr-service` MUST have unit tests for its HTTP handler (invalid input, valid PNG output) and the core rendering logic; boundary tests against a real HTTP server are acceptable per Constitution Principle III

### Key Entities

- **QR Render Request**: `content` (URL string to encode), `size` (integer pixel dimension)
- **QR Render Response**: PNG bytes with `Content-Type: image/png`

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The presenter's `/host` page continues to display a scannable QR code after the migration — observed in a browser against the running compose stack
- **SC-002**: Scanning the QR code with a phone continues to reach the `/play?w=<id>` URL with no additional steps
- **SC-003**: `app/internal/qrcode/qrcode.go`'s `RenderPNG` function is removed from the app module; `go build ./...` in `app/` succeeds without it
- **SC-004**: `docker compose up` from a fresh clone reaches a demoable state with the qr-service running alongside the existing services, without any host-side installs beyond Docker Desktop and git (Constitution Principle II)
- **SC-005**: All existing app tests continue to pass after the migration; the `handleQRPNG` and `handleRepoQRPNG` test doubles are updated to stub the qr-service HTTP call rather than the in-process function

## Assumptions

- The `qr-service` will be a minimal Go HTTP service (consistent with Backend: Go in the Technology Stack) with a single route, mirroring the structure of `commits-service` and `scores-service`
- The service will use `github.com/skip2/go-qrcode` — the same library currently in `app/go.mod` — so no new runtime dependency is introduced (no constitution amendment needed)
- The app service makes a synchronous HTTP call to qr-service; no message queue or async pattern is needed for this extraction
- `BuildPlayURL` may remain in the app module if preferred, since it contains application-level routing logic (the `/play?w=` URL shape); only the PNG rendering moves to the service
- Port 8084 is assigned to qr-service, following 8082 (commits) and 8083 (scores)
- The qr-service does not need its own Redis connection — it is a pure rendering service; the app continues to own the window state and ngrok discovery
- nginx configuration does not need changes for the initial extraction; the browser still reaches `/qr.png` and `/repo-qr.png` through the app service (which now proxies to qr-service internally)
